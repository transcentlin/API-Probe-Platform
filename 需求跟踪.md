# API 平台智能探测系统 — 需求跟踪

本文件用于回顾和跟踪 API 平台智能探测系统自开发以来的新需求引入、原有功能变迁以及缺陷修正历史，确保系统架构与业务逻辑的一致性。

---

## 一、 新需求与新增功能跟踪

### 1. 探测引擎多阶段设计与扩展 (M1 / M2 / M3)
- **需求描述**：支持对第三方 API 平台的连通性、API 格式、模型列表、可用性及具体能力进行多级探测。
- **改动详情**：
  - 设计并实现了 **Stage 0 至 Stage 5** 的链式执行机制。
  - **Stage 0 (Preflight)**：利用 HEAD 请求快速探测平台可达性。
  - **Stage 1 (FormatDetection)**：自动识别 OpenAI, Anthropic, Gemini API 格式。
  - **Stage 2 (EndpointDiscovery)**：发现 `/models` 接口公开的模型列表。
  - **Stage 2.5 (ModelAvailability)**：模型可用性预检，防止无效模型进入后续探测。
  - **Stage 3 (CapabilityProbing)**：八大核心能力探针（基础对话、流式、工具调用、视觉、JSON模式、推理、联网、多轮）。
  - **Stage 4 (LimitsDetection)**：边界与限流探测（待接入多级兜底链路）。
  - **Stage 5 (StabilityAnalysis)**：稳定性分析。

### 2. 超时配置与分级判定设计 (M2+)
- **需求描述**：由于不同大模型响应速度差异巨大，需避免因硬编码超时（如原 30s）导致慢速但可用的模型被误判。
- **改动详情**：
  - 引入了 `ResponseHealth` 状态分级（fast / slow / critical）。
  - 提取超时配置到 `user_config.yaml` 配置文件中（`defaults.timeouts` 下的 `fast_threshold_ms`、`slow_threshold_ms`、`connect_timeout_s`），实现参数免侵入式修改。

### 3. 全探针“非流式 + 流式 fallback”改造 (M2+)
- **需求描述**：许多聚合平台对某些模型的非流式响应支持不佳（例如直接返回 200 空响应），而流式能正常返回。需要全探针具备智能降级和 fallback 机制。
- **改动详情**：
  - 重构了所有 8 个核心探针，当前均支持在非流式请求异常时，自动回退到流式传输探测。
  - 在最终报告中，为“能力矩阵”新增了**「传输通道」**列，清晰展现模型的可用模式（双通道 / 仅流式 🌊 / 仅非流式 ⏳ / 失效 ❌）。

### 4. 语义连贯性判定 (M2+)
- **需求描述**：识别语义崩溃但仍返回 200 成功响应的大模型。
- **改动详情**：
  - 新建 `coherence.py` 模块，通过 5 维启发式算法（重复率、语种突变、生僻字、拉丁突兀、长度异常）对模型输出做内容过滤判定。

### 5. 平台原生 Status 提取与短路熔断 (M2++)
- **需求描述**：避免在平台已经标记为 offline/disabled 的模型上发起多次探测，降低整体探测耗时。
- **改动详情**：
  - 扩展 `UnavailableReason` 枚举，新增 `PLATFORM_DEAD` / `PLATFORM_DEGRADED`。
  - 在 Stage 2 中提取并保留 `/models` 接口返回的原生 status 状态；在 Stage 2.5 对标记不可用的模型直接短路熔断，不再发送探测请求。整体耗时暴降 74%。

### 6. 智能速率限制（RPM）多级自适应探测与来源追溯 (Stage 4)
- **需求描述**：对模型进行黑盒动态速率限制（RPM）探测。不需要且不应在配置文件中静态配置 API 限制，必须纯黑盒动态提取。
- **改动详情**：
  - **第一级**：从 HTTP 响应头中读取常见的标准限流字段（如 `x-ratelimit-limit-requests` 等）。
  - **第二级（端点盲扫）**：账户配额端点路径盲扫，尝试 GET 常用路径（如 `/subscription`、`/key/status`、`/limits`）来提取限流配额。
  - **第三级（429 反推）**：被动 429 报错重置反推，拦截探测过程中的 429 `Retry-After` 间隔，于 Stage 4 进行追溯换算。
  - **第四级（压测判定）**：在 Stage 4 末尾，使用极轻量（极小 token）的瞬时 2 并发与 5 并发温和阶梯压测，估算是否支持高并发，进而给出估算值（如 `≥ 60 RPM` 或 `≥ 300 RPM`）。
  - **来源追溯**：边界限制数据类新增 `rate_limit_source` 字段；在最终报告的“第七部分：边界探测”表格中新加「速率获取来源」一列并配以拟态 Emoji 展示，大幅提升数据透明度。

### 7. 统一加密平台注册表 (Phase 1 & 1.1)
- **需求描述**：将原本分裂的 `user_config.yaml`（明文平台列表）与 `.env.enc`（加密 API 密钥）合并，统一保存在加密的 `config/platforms.enc` 注册表中，并对 CLI 提取通用运行核心，方便以后 Web UI 独立调用。
- **改动详情**：
  - 实现了 `PlatformManager` 提供对平台条目的 CRUD 方法；
  - 升级迁移工具自动合并原先隐藏的 14 个平台，完成全量 18 个平台的一键迁移；
  - 从 `run.py` 剥离了核心探测到独立的 `probe_runner.py` 中，方便多入口共用；
  - 精简了原 `user_config.yaml`，仅保留默认的超时及全局探测档位配置，且在 `.gitignore` 中排除了 `*.enc`，防止敏感配置泄露。

### 8. 非标平台专属方案 B 与预检测端点解耦 (Phase 2)
- **需求描述**：提供专属的模型发现处理器以支持非标平台（如 Ollama 的 `/api/tags`，Cloudflare Workers AI 的 `/ai/models/search` 等）的原生拉取。同时必须解耦可用性预检测（Stage 2.5）中的硬编码对话端点（如原硬编码为 `/chat/completions`），避免非标适配器在预检时抛出 404 故障。
- **改动详情**：
  - 设计并引入了模型发现注册表 `ModelDiscoveryRegistry`，实现 `OpenAIDiscoveryHandler`、`OllamaDiscoveryHandler`、`CloudflareDiscoveryHandler`；
  - 重构了 Stage 2.5 预检，在流式与非流式预检测发包时，动态读取并渲染各适配器 `FormatAdapter` 对应的 `default_endpoint` 以及 headers，使对标准平台 100% 零修改，对 Ollama 平台自适应适配到 `/api/chat` 并正确获取 404 捕获。
  - 修改 Ollama 的 `base_url` 为 `https://ollama.com`，彻底消除冗余的 `/api/api/chat` 错拼 404。

### 9. 多平台横向选型对比历史管理与文件存储 (Phase 2+)
- **需求描述**：支持将多平台对比数据持久化为 `Comparison_Report_*.md` 物理文件，并在前端提供历史对比报告的发现、在线预览、物理下载与删除功能。
- **改动详情**：
  - 后端在 `reports.py` 挂载 `COMPARISON_PATTERN` 正则匹配，在 `/reports/list` 接口中支持以 `platform="横向对比"` 定向获取并按时间戳逆序排序返回，完全省去无关文件的限制。
  - 前端重构 `ComparePage.jsx` 为经典双 Tab 页布局（“最新对比报告”与“历史报告发现”），打通了在线预览、本地下载以及物理删除的逻辑闭环。

## 二、 已有功能修正与缺陷修复跟踪

### 1. 错误细分类增强与“未知”错误清理 (M2+)
- **旧逻辑**：探针抛出的大量异常直接归为 `UNKNOWN`（未知）原因，报告可读性差。
- **修正后**：
  - 扩展 `UnavailableReason` 枚举，新增 `NETWORK_ERROR`、`UPSTREAM_ERROR`、`SERVER_ERROR`、`RESPONSE_INVALID`、`RESPONSE_INCOHERENT` 等精确成因。
  - 补齐了 `ErrorCategory` 到 `UnavailableReason` 的完整映射词典，将 13 类错误细化。

### 2. Socket 超时卡死漏洞防御 (近期修复)
- **旧逻辑**：针对“慢速连接”特征模型（周期性极慢发送极少字节），`httpx` 的底层 socket 读超时器会被重置，导致请求挂死甚至整个系统阻塞卡死。
- **修正后**：
  - 在核心发包器 `base.py`（非流式 `_send_request` 及流式 `_send_streaming_request`）和 `stages.py` 的可用性预检中，强制包裹 `asyncio.wait_for` 协程级超时拦截器，保障无论 socket 如何变化，到达超时时间加缓冲后必须强制退出。
  - 此项改动使得在 BlazeAI 和 OpenCodeZen 深度模式下的探测成功脱困，耗时从几十分钟下降至 100 多秒。

### 3. 配置管理与安全脱敏
- **旧逻辑**：明文 API Key 容易随代码提交到 Git 仓库，且 YAML 语法直接暴露敏感变量。
- **修正后**：
  - 引入 `EnvVault` 模块与 `manage_keys.py` 工具，使用 Fernet 对 `.env.enc` 加密，主密码保存在 Windows 凭据管理器中。
  - 完善了 `.gitignore`，过滤 `.env` / `.env.enc` 等安全敏感文件以及本地临时过程文档，杜绝泄露风险。

### 4. 边界探测档位精简与容错优化 (v1.4)
- **旧逻辑**：边界探测（Stage 4）的 `max_tokens` 候选测试档位偏多（7档），测试成本高，且当测试较大档位报错（如 400 Bad Request）时，代码会直接 `break` 终止探测，导致后续档位也无法被探测，在报告中呈现为“未探测”。
- **修正后**：
  - 将 `max_tokens`（4 档：65536, 16384, 8192, 4096）与 `max_context`（4 档：4000, 16000, 64000, 128000）精简为 4 档典型代表性段位。
  - 优化 `_probe_max_tokens` 与 `_probe_max_context` 的降级测试逻辑：若高档位请求由于参数超限发生非 200 报错或网络异常，改为 `continue` 尝试下一个稍小的候选档位，直到成功，彻底消除了由于大参数报错引发提早中断导致“未探测”的缺陷（如 `grok-4.20-fast` 成功测出了 4,096 的最大输出）。

### 5. 能力矩阵重构与传输通道简化 (v1.4)
- **旧逻辑**：报告包含一列“传输通道”（如“双通道”、“仅流式”），其包含的信息部分与流式重复，造成能力矩阵列数过多、横向排版臃肿。
- **修正后**：
  - 彻底剔除可用模型能力矩阵中的「传输通道」列。
  - 重构列排布顺序，将「基础对话」重命名为更物理直观的「非流式」，与「流式」并排放在矩阵最左侧前两列（`流式` -> `非流式` -> `其他能力`），通过其对应的 `✓` / `✗` 状态在物理上直接表现出复合通道兼容性。
  - 删除了附录 A.2.1 节冗余的「传输通道说明」小结。

### 6. 边界探测通道自适应短路与流式 Fallback (v1.5)
- **旧逻辑**：边界探测强制通过非流式请求发送。对于在 Stage 3 探测出“非流式已死”的仅流式可用模型，直接测试其非流式的 max_tokens 和 max_context 在物理上 100% 失败，从而无意义地浪费探测请求，且报告中仅生硬地展示为“未探测”。
- **修正后**：
  - **自适应短路**：在边界探测开始时识别每个模型的通道可用性。若模型不支持非流式通道，`max_tokens` 探测直接短路跳过，并赋予具体的简短成因 `不支持非流式通道`。
  - **流式 Fallback 上下文探测**：在测试仅流式可用的模型的 `max_context` 时，系统自动 Fallback 切换为流式接口发送长 Prompt，并通过客户端异步读取首包连接状态（status=200）来提取实际的上下文极限。
  - **自适应压测 (RPM)**：针对仅流式可用模型，第四级并发压测自适应使用流式传输机制发送小包请求，不再因不支持非流式接口而导致压测失败。
  - **精细化成因报告**：`ModelLimits` 字段类型扩展为支持 `str`。若探测超时、通道不支持或彻底失效，在最终报告的边界探测表格中展示具体倾斜成因（如 `_不支持非流式通道_`、`_探测超时/异常_`、`_仅流式-探测超时/异常_`），极大提高了数据的合理性与清晰度。

### 7. 磁盘 I/O 阻塞调优与零 I/O 缓存设计 (v1.5)
- **旧逻辑**：在云盘同步环境下，多平台并行扫描以及报告详情预览高频调用磁盘 stat 探针，带来极大的 I/O 阻塞。
- **修正后**：
  - 对 `list_platforms` 进行单次扫描优化，在内存构建 O(1) 映射表，将 $N$ 次 stat 磁盘高频调用减少为 $1$ 次。
  - 对 `/reports/latest` and `/reports/{filename}` 读取接口引入内存缓存机制，命中缓存直接返回 string，省去高频的 `stat().st_mtime` 系统调用，实现预览报告秒开。

### 8. 彻底移除硬编码人工加载锁 (v1.5)
- **旧逻辑**：为了首屏加载和刷新的“过渡仪式感”，硬编码了 1.5s 的加载静态时延锁，导致网络良好的状态下操作依然觉得卡顿。
- **修正后**：
  - 彻底移除了 `fetchPlatforms` 逻辑中人为设置的 1.5 秒加载延时锁，数据拉取与首屏响应瞬开。

### 9. Socket 慢速超时卡死防御协程 (v1.5)
- **旧逻辑**：遇到恶意在长周期内极慢发送极少字节的慢速大模型时，会不断重置 socket 的底层超时计数器，导致发包挂死系统阻塞。
- **修正后**：
  - 在核心发包器 `base.py`（非流式与流式）和 `stages.py` 中强制使用 `asyncio.wait_for` 超时硬拦截器，保障到达超时时间加缓冲后必须强制断开。
