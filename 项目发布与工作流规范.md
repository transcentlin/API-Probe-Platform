# API 平台测试系统 - 项目发布与工作流规范

本规范定义了每次代码迭代后，AI 在进行仓库更新、源码打包与 GitHub Release 发布时必须遵循的“四阶段自动化标准流水线”。

---

## 执行前提校验
在启动发布流水线前，AI 必须确认：
1. 本地已成功通过 `pytest` 自动化测试。
2. 本地已通过 `gh auth status` 确认 GitHub CLI 处于登录状态。

---

## 第一阶段：本地文件安全物理清理
AI 必须根据 [项目打包与清理指南.md](项目打包与清理指南.md) 的规则，物理切除本地的临时开发垃圾和测试缓存（C 类文件）：

1. **执行删除**：
   In PowerShell 中运行以下物理删除指令：
   ```powershell
   Remove-Item -Path ".pytest_cache", ".claude", "Version Control", "scratch", "api-probe-system_V*.zip" -Recurse -Force -ErrorAction SilentlyContinue
   ```
2. **特殊保留核对**：
   确保 B 类资产（如 `API Probe.exe`、`platforms.enc`、敏感文档等）依然完好地保留在本地。

---

## 第二阶段：Git 状态核实与本地提交
1. **添加受追踪代码**：
   将发生变动的核心源码和配置文件（A 类）添加到暂存区：
   ```bash
   git add api_probe_system/ tests/ run_web.py .gitignore LICENSE README.md 项目发布与工作流规范.md
   ```
   *注意：决不能执行 `git add .` 以防漏过未在忽略列表里的敏感文件。*
2. **核查 Git 状态**：
   运行 `git status` 确认待提交的文件中没有混入 `platforms.enc`、.exe 或其他 B 类私密凭据。
3. **本地规范 Commit**：
   采用 Conventional Commits 格式创建提交：
   ```bash
   git commit -m "<根据修改生成的 Conventional 提交信息>"
   ```

---

## 第三阶段：项目源码安全打包 (ZIP)
为防止将 `node_modules` 依赖或密钥误打包，必须使用“系统临时中转区隔离打包”的策略：

1. **建立纯净临时区**：
   ```powershell
   # 1. 创建临时区
   New-Item -ItemType Directory -Path "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp" -Force
   New-Item -ItemType Directory -Path "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp\reports" -Force

   # 2. 拷贝核心发布文件（排除本地专属清理指南及 Version Control）
   Copy-Item "api_probe_system" "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp\" -Recurse -Force
   Copy-Item "tests" "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp\" -Recurse -Force
   Copy-Item "run_web.py", "LICENSE", "README.md", ".gitignore" "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp\" -Force
   Copy-Item "reports\报告模板规范.md" "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp\reports\" -Force

   # 3. 物理切除依赖与密钥
   Remove-Item -Path "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp\api_probe_system\frontend\node_modules" -Recurse -Force -ErrorAction SilentlyContinue
   Remove-Item -Path "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp\api_probe_system\config\platforms.enc" -Force -ErrorAction SilentlyContinue
   ```
2. **确定发布版本号**（基于上一次发布的 Git Tag）：
   * **修订版 (v1.0.x)**：如果仅包含 Bug 修复。
   * **次版本 (v1.1.x)**：如果引入了向下兼容的功能性新增。
3. **打包压缩**：
   在 PowerShell 中运行压缩，将 zip 文件生成在工作区根目录下，命名为 `api-probe-system_V<版本号>.zip`。
   ```powershell
   Compress-Archive -Path "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp\*" -DestinationPath "D:\同步盘\BaiduSyncdisk\Antigravity\ProjectAsset\API平台测试 260601\api-probe-system_V<版本号>.zip" -Force
   # 物理删除临时区
   Remove-Item -Path "C:\Users\Transcent\AppData\Local\Temp\api_probe_build_temp" -Recurse -Force
   ```

---

## 第四阶段：自动推送代码并发布云端 Release
1. **解除环境变量冲突**（**【极重要】** 必须清空系统注入的无效 `GITHUB_TOKEN`，以便 `gh` 客户端采信用户本机的持久化登录凭据）：
   ```powershell
   $env:GITHUB_TOKEN = $null
   ```
2. **推送代码**：
   ```bash
   git push origin main
   ```
3. **自动发布云端 Release**：
   调用 GitHub CLI 创建 Release 标签，并自动将生成的纯净打包 ZIP 作为 Assets 附件上传发布：
   ```bash
   gh release create v<版本号> api-probe-system_V<版本号>.zip --title "API Probe Platform v<版本号>" --notes "<本次版本的改动说明记录>"
   ```
