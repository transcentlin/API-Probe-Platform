本文档将项目《执行计划.md》正式展开为标准**需求规格说明书（Software Requirements Specification, SRS）**，明确「API平台智能探测系统」**第一阶段（Windows 桌面版）**需要实现的功能需求与非功能需求，作为后续设计、开发、测试、验收的统一依据。

## 2. 总体描述

### 2.1 产品定位

「API平台智能探测系统」是一个**API 平台智能探测与画像工具**。其本质是：

> 用户只提供**最小信息**（API 名称/地址/密钥/官网），系统通过自动化测试**探测并发现**每个 API 平台及其模型的全部信息（格式、模型、能力、限制、性能、稳定性、错误模式），生成完整能力画像，帮助用户**精准选型**。

**核心理念对比**：

| 维度 | ❌ 验证式测试（非本系统） | ✅ 探索式发现（本系统） |
|------|--------------------------|------------------------|
| 用户输入 | 完整配置（格式、模型、参数全填） | 仅 4 项最小信息 |
| 系统职责 | 验证用户配置是否正确 | 自动探测、发现、生成画像 |
| 根本假设 | 用户清楚 API 一切细节 | 用户对 API 细节**一概不清楚** |

### 2.2 用户画像（Persona）

**典型用户**：注册了多个（约 10 个）AI API 平台的开发者 / 重度 AI 工具使用者。

- **行为**：在 Cherry Studio、Claude Code、Codex 等工具中接入各平台 API
- **痛点**：频繁踩坑 —— 模型突然不可用、只支持特定格式、不支持工具调用/JSON 等；每次接入靠试错；缺乏系统化测试与文档
- **诉求**：有一个工具能自动把每个平台"测明白"，输出可直接用于选型的清晰画像

### 2.3 核心理念与根本假设

**根本假设**：用户能稳定提供的，只有 4 项基础信息：

1. `name` —— API 平台名称（必填）
2. `base_url` —— API 地址（必填）
3. `api_key` —— API 密钥（必填，支持环境变量引用）
4. `website` —— API 官网（可选，用于抓取文档线索）

其余一切（API 格式、可用模型、能力矩阵、限制、性能、稳定性、错误模式）**均由系统探测产出**，不要求用户预先知晓。

### 2.4 真实输入数据特征

系统首批面向的 10 个真实平台（来自 `API平台列表.md`），其特征决定了需求的具体性：

| 特征 | 实际情况 | 对需求的影响 |
|------|---------|-------------|
| 平台数量 | 10 个（Grok2API、FreetheAI、BlazeAI、Nvidia、GitHub、Cerebras、Groq、OpenRouter、Gemini、Kiro） | 需支持批量配置与批量探测 |
| API 格式 | 3 种：多数为 OpenAI Chat Completions；Kiro 为 Anthropic Messages；Gemini 为 Gemini Native（并存 OpenAI 兼容端点） | 格式识别必须覆盖 3 种并能处理"多格式兼容" |
| 地址形态 | 既有公网 https，也有本地 `http://localhost:8000`、`http://localhost:8088/` | 预检查须兼容本地与远程 |
| 密钥形态 | 含 `${ENV}` 引用、多密钥并列、带 `@` 注释等 | 配置解析须健壮，密钥须脱敏保护 |
| 已知问题样本 | Qwen 工具调用失败、DeepSeek `Provider_failed`、kimi `HTTP200 假成功`、Grok 只能聊天不能用工具 | 能力探针 + 错误模式识别须能暴露这些真相 |

### 2.5 假设、依赖与约束

- **假设**：用户提供的密钥在测试时有效且有可用额度；测试环境可访问目标平台（含本地服务已启动）
- **依赖**：Python 3.11+ 运行环境；网络连通；目标平台在线
- **约束**：
  - 边界探测（Stage 4）会消耗较多请求与额度，**必须可开关、可限额**
  - 单次文档/配置写入受平台限制（参见工程约定），不影响运行期需求
  - 第一阶段仅交付 Windows 桌面版

## 4. 功能需求（按功能模块分组）

> 体例说明：每个模块包含【模块目标】→【用户故事】→【功能需求条目 FR】→【验收标准】。FR 编号格式 `FR-<模块>-<序号>`，全局唯一，供第 8 章追溯。

### 4.1 M1 配置管理（`FR-CFG-xx`）

**模块目标**：让用户以最小成本录入和维护平台清单，安全解析密钥，并在探测前校验配置基本可用。

**用户故事**
- **US-CFG-1**：作为用户，我想只填 4 项最小信息就能录入一个平台，以便不必预先了解 API 细节。
- **US-CFG-2**：作为用户，我想用环境变量引用密钥（如 `${GROQ_API_KEY}`），以便不在配置文件里明文存放密钥。
- **US-CFG-3**：作为用户，我想增删改平台并在录入时得到基本校验，以便尽早发现填写错误。

**功能需求**

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| FR-CFG-01 | 系统应支持录入平台的最小信息：`name`(必填)、`base_url`(必填)、`api_key`(必填)、`website`(可选)、`notes`(可选) | 必须 |
| FR-CFG-02 | 系统应支持可选的 `hints`（如 `api_format`、`models_endpoint`），留空时由系统自动探测 | 必须 |
| FR-CFG-03 | 系统应支持 `api_key` 的环境变量引用语法 `${VAR_NAME}`，并在运行时从环境解析真实值 | 必须 |
| FR-CFG-04 | 系统应支持平台的新增、编辑、删除、批量导入（从 `user_config.yaml`） | 必须 |
| FR-CFG-05 | 系统应在录入/保存时校验：name 非空且唯一、base_url 为合法 URL、api_key 非空 | 必须 |
| FR-CFG-06 | 系统应兼容本地地址（`http://localhost:PORT`）与公网地址（`https://`） | 必须 |
| FR-CFG-07 | 配置持久化应区分两层：用户输入层 `user_config.yaml` 与系统发现层 `discovered_profile.yaml`，二者不混写 | 必须 |

**验收标准**
- **AC-CFG-01**：Given 用户仅填写 name/base_url/api_key，When 保存配置，Then 配置成功且不强制要求其它字段。
- **AC-CFG-02**：Given api_key 填写为 `${GROQ_API_KEY}` 且该环境变量已设置，When 系统读取配置，Then 解析得到真实密钥；若环境变量缺失，Then 给出明确错误提示而非静默失败。
- **AC-CFG-03**：Given 用户填写非法 base_url（如缺少协议头），When 保存，Then 系统拒绝保存并提示具体原因。
- **AC-CFG-04**：Given 已存在同名平台，When 再次以相同 name 录入，Then 系统提示名称冲突。

### 4.2 M2 探测引擎编排（`FR-PROBE-xx`）

**模块目标**：以渐进式 6 阶段编排整个探测流程，支持 4 种探测模式，保证任一阶段失败不中断整体流程。

**用户故事**
- **US-PROBE-1**：作为用户，我想选择探测深度（快速/标准/深度/自定义），以便在耗时与信息量之间权衡。
- **US-PROBE-2**：作为用户，我想即使某阶段失败，系统也能记录原因并继续后续可执行的探测，以便一次跑出尽可能多的信息。
- **US-PROBE-3**：作为用户，我想每个阶段的产出作为下一阶段的输入被自动衔接，以便无需人工干预。

**功能需求**

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| FR-PROBE-01 | 系统应按固定顺序编排 6 阶段：Stage0 预检查 → Stage1 格式识别 → Stage2 端点发现 → Stage3 能力探测 → Stage4 边界探测 → Stage5 稳定性 | 必须 |
| FR-PROBE-02 | 系统应实现 Stage0 预检查：校验 base_url 合法性、网络连通性（DNS/TCP/TLS）、api_key 非空格式；可选抓取 website 提取文档线索 | 必须 |
| FR-PROBE-03 | 系统应实现 Stage1 格式识别（详见 M3），输出格式与置信度 | 必须 |
| FR-PROBE-04 | 系统应实现 Stage2 端点发现：依次尝试 `/models`、`/v1/models`、`/api/models` 等，解析并提取全部 model id | 必须 |
| FR-PROBE-05 | 系统应实现 Stage3 能力探测（详见 M4），对每个发现的模型逐项测试能力 | 必须 |
| FR-PROBE-06 | 系统应实现 Stage4 边界探测：上下文窗口、最大输出、速率限制、并发限制、超时行为；该阶段必须可开关、可限额 | 必须 |
| FR-PROBE-07 | 系统应实现 Stage5 稳定性：一致性测试、间歇性故障检测、错误模式识别、稳定性评分（详见 M5） | 必须 |
| FR-PROBE-08 | 任一阶段失败时，系统应记录失败原因，不中断流程，继续执行后续可独立运行的阶段 | 必须 |
| FR-PROBE-09 | 系统应支持 4 种探测模式：快速(Stage0-2)、标准(Stage0-3，默认)、深度(Stage0-5)、自定义(任选阶段/模型/能力) | 必须 |
| FR-PROBE-10 | 每个阶段的产出应结构化记录，并作为后续阶段的输入（如 Stage2 的模型列表喂给 Stage3） | 必须 |

**验收标准**
- **AC-PROBE-01**：Given 选择"标准测试"，When 启动探测，Then 系统依次执行 Stage0–Stage3 且不执行 Stage4–5。
- **AC-PROBE-02**：Given Stage1 格式识别失败，When 继续探测，Then 系统记录失败原因，并仍尝试 Stage2 端点发现等后续阶段（在可行范围内）。
- **AC-PROBE-03**：Given 选择"深度测试"且未开启边界限额，When 进入 Stage4，Then 系统按配置的额度上限执行并在达到上限时停止该项探测。
- **AC-PROBE-04**：Given 选择"自定义"并仅勾选某平台的"工具调用"探针，When 启动，Then 系统只执行该探针，跳过其它能力测试。

### 4.3 M3 格式适配（`FR-ADAPT-xx`）

**模块目标**：自动判断每个平台属于哪种 API 格式，输出置信度，并以插件化适配器屏蔽格式差异，支持识别"多格式兼容"平台。

**用户故事**
- **US-ADAPT-1**：作为用户，我完全不知道某平台是什么格式，我想系统自动告诉我它是 OpenAI / Anthropic / Gemini 中的哪一种以及有多确定。
- **US-ADAPT-2**：作为用户，对于像 Gemini 这种同时提供原生与 OpenAI 兼容端点的平台，我想系统记录它兼容多种格式。

**功能需求**

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| FR-ADAPT-01 | 系统应内置三种格式适配器：OpenAI Chat Completions、Anthropic Messages、Gemini Native，且以插件形式组织，新增格式只需加适配器 | 必须 |
| FR-ADAPT-02 | 系统应对每种格式发送最小探针请求，依据 HTTP 状态码 + 响应体结构特征打分 | 必须 |
| FR-ADAPT-03 | 系统应取最高分格式输出 `detected`，并给出 `confidence`（0–1） | 必须 |
| FR-ADAPT-04 | 系统应记录所有尝试过的格式及其结果（`alternatives_tested`），以识别多格式兼容平台 | 必须 |
| FR-ADAPT-05 | 格式判定的响应特征应至少包含：OpenAI `choices[].message.content`；Anthropic `content[].type/text` 及顶层 `role`；Gemini `candidates[].content.parts[]` | 必须 |
| FR-ADAPT-06 | 若用户在 `hints.api_format` 提供了格式提示，系统应优先验证该提示，但仍记录实际探测结果 | 应该 |

**验收标准**
- **AC-ADAPT-01**：Given 一个 OpenAI 兼容平台，When 执行格式识别，Then `detected=openai_chat_completions` 且 confidence ≥ 0.8。
- **AC-ADAPT-02**：Given Kiro（Anthropic Messages），When 识别，Then `detected=anthropic_messages`。
- **AC-ADAPT-03**：Given Gemini（同时支持原生与 OpenAI 兼容），When 识别，Then `alternatives_tested` 中记录了不止一种成功格式。

### 4.4 M4 能力探针（`FR-CAP-xx`）

**模块目标**：对每个发现的模型，逐项、独立地测试其 8 项能力，每项独立记录"成功/失败/不支持"。

**用户故事**
- **US-CAP-1**：作为用户，我想知道每个模型到底能不能调用工具、能不能看图、能不能输出 JSON，以便按需选型。
- **US-CAP-2**：作为用户，我想每项能力测试相互独立，某项失败不影响其它项的判定。

**功能需求**

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| FR-CAP-01 | 系统应对每个模型执行"基础对话"探针，验证模型基本可用 | 必须 |
| FR-CAP-02 | 系统应执行"流式响应"探针（`stream=true`），验证 SSE 流式输出 | 必须 |
| FR-CAP-03 | 系统应执行"工具调用"探针，发送 tool/function 定义，验证是否正确返回 tool_call | 必须 |
| FR-CAP-04 | 系统应执行"视觉理解"探针，分别以图片 URL 与 base64 两种方式发送，验证多模态 | 必须 |
| FR-CAP-05 | 系统应执行"JSON 模式"探针，要求结构化输出，验证 json_mode / json_schema | 必须 |
| FR-CAP-06 | 系统应执行"推理能力"探针，检测响应中是否含 reasoning/thinking 字段 | 应该 |
| FR-CAP-07 | 系统应执行"联网搜索"探针，发送需实时信息的问题，检测是否联网 | 应该 |
| FR-CAP-08 | 系统应执行"多轮对话"探针，验证上下文保持 | 必须 |
| FR-CAP-09 | 每项探针应独立记录结果：`supported`、`reliability`、`failed_models`、`tested_at` | 必须 |
| FR-CAP-10 | 能力探针应以插件形式组织，可单独增删某项探针 | 应该 |

**验收标准**
- **AC-CAP-01**：Given 一个模型，When 执行 8 项探针，Then 每项产出独立的 supported/不支持/失败结论与时间戳。
- **AC-CAP-02**：Given Grok（已知只能聊天不能用工具），When 执行工具调用探针，Then 结果标记为"不支持/失败"，而基础对话探针仍为"支持"。
- **AC-CAP-03**：Given Qwen（已知工具调用失败），When 执行工具调用探针，Then 系统如实记录失败而非误判为成功。
- **AC-CAP-04**：Given 视觉探针，When 同时尝试 URL 与 base64，Then 两种方式的结果分别记录。

### 4.5 M5 结果分析与评分（`FR-ANALYZE-xx`）

**模块目标**：基于探测原始数据，识别错误模式、计算稳定性评分、生成选型推荐，**暴露隐蔽真相**。

**用户故事**
- **US-ANALYZE-1**：作为用户，我想系统能识别"HTTP200 但实际失败""Provider_failed"这类隐蔽问题，以免被假成功误导。
- **US-ANALYZE-2**：作为用户，我想得到每个平台/模型的稳定性评分与"适合/不适合"推荐，以便直接决策。

**功能需求**

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| FR-ANALYZE-01 | 系统应识别错误模式，至少覆盖：HTTP200 假成功（如 kimi）、Provider_failed（如 DeepSeek）、工具调用格式错误（如 Qwen） | 必须 |
| FR-ANALYZE-02 | 每个错误模式应记录 `pattern`、`affected_models`、`description` | 必须 |
| FR-ANALYZE-03 | 系统应执行一致性测试（相同请求重复 N 次）与间歇性故障检测（统计 uptime/error rate） | 必须 |
| FR-ANALYZE-04 | 系统应综合计算稳定性评分（0–1），含 `score`、`uptime_rate`、`error_rate`、`tested_requests` | 必须 |
| FR-ANALYZE-05 | 系统应测量并记录性能指标：`avg_latency_ms`、`ttfb_ms`、`tokens_per_second`、`success_rate` | 必须 |
| FR-ANALYZE-06 | 系统应生成推荐：`best_for`、`avoid_for`、`overall_rating` | 必须 |
| FR-ANALYZE-07 | 分析结果应写入 `discovered_profile`，作为报告生成的数据源 | 必须 |

**验收标准**
- **AC-ANALYZE-01**：Given kimi 返回 HTTP200 但响应体为错误，When 分析，Then 系统标记为"HTTP200 假成功"错误模式而非成功。
- **AC-ANALYZE-02**：Given 某模型多次请求中部分失败，When 计算稳定性，Then 输出 0–1 的 score 及对应 uptime/error rate。
- **AC-ANALYZE-03**：Given 完成探测的平台，When 生成推荐，Then 给出"适合场景/不适合场景/综合评分"。

### 4.6 M6 报告生成（`FR-REPORT-xx`）

**模块目标**：将能力画像转化为可直接用于选型决策的报告，支持多类型、多格式导出。

**用户故事**
- **US-REPORT-1**：作为用户，我想一键生成报告并保存到本地，以便归档和随时查阅。
- **US-REPORT-2**：作为用户，我想报告能"可直接行动"，明确告诉我"想要工具调用稳定 → 用哪个平台的哪个模型"。

**功能需求**

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| FR-REPORT-01 | 系统应支持 4 类报告：综合报告、单平台报告、单模型报告、对比报告 | 必须（综合+单平台为必须，其余应该） |
| FR-REPORT-02 | 第一阶段应至少导出 Markdown 与 HTML 两种格式；JSON、Excel 为可选 | 必须(MD+HTML) / 应该(JSON+Excel) |
| FR-REPORT-03 | 综合报告应含：执行摘要（成功/失败数、模型总数、平均速度、整体稳定性、推荐 Top3）、每平台详细结果、对比分析、附录 | 必须 |
| FR-REPORT-04 | 每平台详细结果应含：基本信息(URL/格式/置信度)、模型列表表格(工具/视觉/JSON/速度/状态)、能力矩阵、性能指标、限制、已知问题、推荐场景、评分 | 必须 |
| FR-REPORT-05 | 报告应明确高亮隐蔽问题（如"HTTP200 但实际失败"） | 必须 |
| FR-REPORT-06 | 报告应保存到 Session 工作目录，不得留在临时目录 | 必须 |
| FR-REPORT-07 | JSON 报告应为机器可读，便于其他工具消费（如生成 Cherry Studio 配置） | 应该 |

**验收标准**
- **AC-REPORT-01**：Given 完成探测，When 生成综合报告，Then 输出含执行摘要与每平台详细结果的 MD 与 HTML 文件并落盘到 Session 目录。
- **AC-REPORT-02**：Given 某平台存在 HTTP200 假成功问题，When 查看报告，Then 该问题被显著标注。
- **AC-REPORT-03**：Given 报告导出，When 打开 HTML，Then 关键信息（格式、模型、能力、稳定性）清晰可读。

### 4.7 M7 任务调度（`FR-TASK-xx`）

**模块目标**：支持全自动一键探测与手动精细触发，控制并发与额度，实时上报进度，支持暂停/停止。

**用户故事**
- **US-TASK-1**：作为用户，我想选定平台后一键跑完全部阶段，无需逐步干预。
- **US-TASK-2**：作为用户，我想手动勾选特定平台/模型/测试项单独触发，以便精准补测。
- **US-TASK-3**：作为用户，我想控制边界探测的额度消耗，避免烧光 API 额度。

**功能需求**

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| FR-TASK-01 | 系统应支持全自动模式：选定一个或多个平台后一键跑完所选探测模式的全部阶段 | 必须 |
| FR-TASK-02 | 系统应支持手动触发：勾选特定平台 / 特定模型 / 特定测试项，单独执行 | 必须 |
| FR-TASK-03 | 系统应支持多平台/多探针的并发执行，并提供并发上限配置 | 必须 |
| FR-TASK-04 | 系统应实时上报进度：当前平台、当前阶段、阶段内进度百分比、整体平台进度（如 2/5） | 必须 |
| FR-TASK-05 | 系统应支持暂停、继续、停止探测任务 | 必须 |
| FR-TASK-06 | 系统应提供额度/请求数限额开关，尤其约束 Stage4 边界探测的消耗 | 必须 |
| FR-TASK-07 | 系统应支持断点续测：中断后可从未完成的阶段/平台继续，已完成结果不重跑 | 应该 |

**验收标准**
- **AC-TASK-01**：Given 勾选 3 个平台并选"标准测试"，When 点击开始，Then 系统自动依次/并发完成 3 个平台的探测并实时显示进度。
- **AC-TASK-02**：Given 仅勾选某平台某模型的"工具调用"项，When 触发，Then 只执行该项测试。
- **AC-TASK-03**：Given 设置边界探测请求上限为 N，When Stage4 达到 N 次请求，Then 停止该项并记录"因限额停止"。
- **AC-TASK-04**：Given 任务运行中，When 点击停止，Then 任务在安全点终止并保留已得结果。

### 4.8 M8 桌面界面（`FR-UI-xx`）

**模块目标**：提供 Windows 桌面单窗口界面，覆盖配置、控制、监控、结果与报告导出全流程（对应执行计划 5.2 线框）。

**用户故事**
- **US-UI-1**：作为用户，我想在一个窗口内完成"选平台→设模式→启动→看进度→出报告"全过程。
- **US-UI-2**：作为用户，我想实时看到探测日志逐条滚动，以便了解系统正在做什么。

**功能需求**

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| FR-UI-01 | 界面应提供左侧平台清单：勾选 + 新增/编辑/删除 | 必须 |
| FR-UI-02 | 界面应提供测试控制区：探测模式单选(快速/标准/深度/自定义)、范围多选(连通/格式/能力/边界等)、开始/暂停/停止按钮 | 必须 |
| FR-UI-03 | 界面应提供测试进度区：当前平台、当前阶段(如 能力 3/6)、进度条、平台计数(如 2/5) | 必须 |
| FR-UI-04 | 界面应提供实时日志区：带时间戳、状态图标(✓/⚠/✗)的逐条日志 | 必须 |
| FR-UI-05 | 界面应提供快速结果条：单行展示某平台的 格式/模型数/稳定性星级 | 必须 |
| FR-UI-06 | 界面应提供报告导出入口：生成报告、导出画像配置、查看详情 | 必须 |
| FR-UI-07 | 界面不得包含业务逻辑，仅通过本地 API/服务调用核心引擎 | 必须 |
| FR-UI-08 | 密钥在界面显示时应脱敏（如仅显示前后若干位） | 必须 |
| FR-UI-09 | 【第二阶段，本期不展开】Web 仪表板、详情页、对比页、配置页等丰富可视化 | — |

**验收标准**
- **AC-UI-01**：Given 主界面，When 用户勾选平台并选模式后点击开始，Then 进度区与日志区实时更新。
- **AC-UI-02**：Given 探测进行中，When 某探针失败，Then 日志区以 ⚠/✗ 图标显示该失败条目。
- **AC-UI-03**：Given 探测完成，When 点击生成报告，Then 报告落盘并提示路径。
- **AC-UI-04**：Given 界面展示已配置平台，When 查看密钥字段，Then 密钥为脱敏显示。

## 5. 非功能需求（NFR）

> NFR 编号格式 `NFR-<类别>-<序号>`，同样纳入第 8 章追溯。

### 5.1 性能（NFR-PERF）

| 编号 | 需求描述 |
|------|---------|
| NFR-PERF-01 | 探测耗时应分档可控：快速探测 ≈5 分钟、标准测试 ≈15 分钟、深度测试 ≈30 分钟+（均为单平台量级参考值） |
| NFR-PERF-02 | 系统应支持并发探测以缩短多平台总耗时，并发度可配置 |
| NFR-PERF-03 | 界面在探测进行时应保持响应（不卡死），日志与进度刷新延迟应可接受（≤1s 量级） |

### 5.2 可靠性（NFR-REL）

| 编号 | 需求描述 |
|------|---------|
| NFR-REL-01 | 阶段隔离：任一探测阶段失败不得导致整个流程崩溃（对应 FR-PROBE-08） |
| NFR-REL-02 | 系统应对网络超时、连接失败等异常进行捕获与重试（有限次），并如实记录 |
| NFR-REL-03 | 断点续测：异常中断后应能从未完成处继续（对应 FR-TASK-07） |
| NFR-REL-04 | 探测结果应及时持久化，避免进程退出导致已得数据丢失 |

### 5.3 安全（NFR-SEC）

| 编号 | 需求描述 |
|------|---------|
| NFR-SEC-01 | 密钥优先以环境变量引用，配置文件中不强制明文存放（对应 FR-CFG-03） |
| NFR-SEC-02 | 密钥在界面、日志、报告中一律脱敏，不得输出完整明文 |
| NFR-SEC-03 | 报告与画像文件中不得包含可还原的完整密钥 |
| NFR-SEC-04 | 原始请求/响应数据若含敏感信息，存储时应做脱敏或访问控制 |

### 5.4 易用性（NFR-USE）

| 编号 | 需求描述 |
|------|---------|
| NFR-USE-01 | 最小输入：用户录入一个平台不应被强制要求填写 4 项以外的信息 |
| NFR-USE-02 | 错误提示应清晰可行动（如"环境变量 X 未设置"而非泛化报错） |
| NFR-USE-03 | 报告应"可直接行动"，给出明确选型建议 |

### 5.5 可维护性（NFR-MAINT）

| 编号 | 需求描述 |
|------|---------|
| NFR-MAINT-01 | 业务逻辑与界面彻底解耦：核心引擎可独立通过 CLI/API 调用，不依赖任何 UI |
| NFR-MAINT-02 | 格式适配器插件化：新增 API 格式只需新增一个适配器，不改动核心编排 |
| NFR-MAINT-03 | 能力探针插件化：每个探针为独立模块，可单独增删 |
| NFR-MAINT-04 | 平台、测试项、探测策略应配置驱动，不在代码中写死 |

### 5.6 可移植性（NFR-PORT）

| 编号 | 需求描述 |
|------|---------|
| NFR-PORT-01 | 第一阶段桌面界面应采用"本地服务 + 界面调 API"形态（建议本地 FastAPI），为第二阶段 Web 化预留路径 |
| NFR-PORT-02 | 分层架构（表示层/接口层/核心引擎/数据层）中，第二阶段 Web 化应只需替换表示层，下三层原样复用 |
| NFR-PORT-03 | 核心后端应运行于 Python 3.11+，跨 Windows 与后续服务器环境可移植 |

### 5.7 成本与额度控制（NFR-COST）

| 编号 | 需求描述 |
|------|---------|
| NFR-COST-01 | 边界探测（Stage4）必须可整体关闭 |
| NFR-COST-02 | 系统应提供请求数/额度上限设置，达到上限即停止相应探测（对应 FR-TASK-06） |
| NFR-COST-03 | 各探测模式应明确告知预计的请求量级，便于用户预估消耗 |

## 7. 外部接口需求

### 7.1 输入接口
- **平台清单输入**：来自 `【重要】API平台列表.md` / `user_config.yaml`，10 个平台、3 种 API 格式
- **目标 API 接口**：被探测的第三方 API（OpenAI Chat Completions / Anthropic Messages / Gemini Native 三种规范），含公网与本地 localhost 端点
